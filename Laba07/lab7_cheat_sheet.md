# ЛР7 Django: шпаргалка по работе и коду

## 1) Что реализовано по варианту 1

Тема: Реорганизация учета основных средств.

Сделано:
- Добавлен новый реквизит ОС: `internal_code` (Код внутреннего учета).
- Код формируется по правилу:
  - `КодПодразделения_ПорядковыйНомер`
  - пример: `IT_1`, `IT_2`, `BUH_1`.
- В форме ОС есть кнопка "Назначить внутренний код".
- При сохранении нового ОС код назначается автоматически.

Как это работает по шагам:
- Пользователь открывает форму создания ОС.
- Выбирает подразделение, вводит название, дату, стоимость и износ.
- Нажимает кнопку назначения кода или просто сохраняет форму.
- Модель `FixedAsset` находит максимальный номер в выбранном подразделении.
- Следующее значение увеличивается на 1.
- В поле `internal_code` записывается итоговый код вида `КодПодразделения_Номер`.

Почему это удобно:
- Пользователь не вводит код вручную и не ошибается в нумерации.
- Код всегда связан с подразделением и отражает порядок добавления ОС.
- Логику можно проверить на любой новой записи без ручного редактирования БД.

## 4) Ключевые модели и методы

### Department (подразделение)
Файл: `assets/models.py`

Поля:
- `code` — код подразделения (уникальный)
- `name` — название
- `description` — описание

### FixedAsset (основное средство)
Файл: `assets/models.py`

Поля:
- `name` — наименование ОС
- `department` — место хранения (FK на Department)
- `cost` — стоимость
- `depreciation_percent` — износ, %
- `sequence_number` — порядковый номер в подразделении
- `internal_code` — код внутреннего учета (уникальный)

Методы:
- `generate_internal_code()`
  - Находит max `sequence_number` в выбранном подразделении.
  - Увеличивает на 1.
  - Формирует `internal_code`.
  - Используется при создании новой записи и при нажатии кнопки в форме.
- `clean()`
  - Проверяет: стоимость > 0.
  - Проверяет: износ от 0 до 100.
  - Запускается до сохранения объекта и не дает записать некорректные данные.
- `save()`
  - Если `internal_code` пуст, назначает автоматически.
  - Вызывает `full_clean()` перед сохранением.
  - Это основная точка контроля перед записью в базу данных.
- `residual_cost` (property)
  - Считает остаточную стоимость после износа.
  - Используется только для отображения, в базу не сохраняется.

Как читать модель:
- `CharField`, `IntegerField`, `DecimalField`, `ForeignKey` описывают поля таблицы БД.
- `ForeignKey` связывает ОС с подразделением, чтобы одно подразделение могло содержать много ОС.
- `unique=True` защищает от одинаковых кодов и дублирования ключевых значений.

Формула:

$$
\text{residual\_cost} = \text{cost} \cdot \left(1 - \frac{\text{depreciation\_percent}}{100}\right)
$$

## 5) Ключевые формы

Файл: `assets/forms.py`

- `DepartmentForm`
  - Нормализует `code` в верхний регистр через `clean_code()`.
  - Это нужно, чтобы коды подразделений выглядели одинаково и не зависели от регистра ввода.

- `FixedAssetForm`
  - Проверяет корректность стоимости (`clean_cost`).
  - Проверяет диапазон износа (`clean_depreciation_percent`).
  - Проверяет, что дата покупки не в будущем (`clean`).
  - Отвечает за первичную защиту от ошибочного ввода пользователя.

Что происходит в форме:
- Django сначала строит HTML на основе формы.
- Потом при отправке берет данные из `POST`.
- После этого выполняются методы `clean_*` и `clean`.
- Если есть ошибки, форма возвращается на страницу с сообщениями.

## 6) Ключевые views

Файл: `assets/views.py`

- `index` — главная страница со статистикой.
- `DepartmentListView/DetailView/CreateView/UpdateView/DeleteView` — CRUD подразделений.
- `FixedAssetListView/DetailView/CreateView/UpdateView/DeleteView` — CRUD ОС.
- `assign_internal_code` — JSON/AJAX endpoint для переназначения кода.

Логика страницы ОС:
- `FixedAssetListView` показывает список, поиск и фильтр по подразделению.
- `FixedAssetDetailView` открывает карточку выбранного ОС.
- `FixedAssetCreateView` и `FixedAssetUpdateView` используют одну и ту же форму.
- При сохранении в `form_valid()` можно показать сообщение об успехе через `messages`.

Логика сообщения и переназначения кода:
- В шаблоне кнопка добавляет скрытое поле `assign_code=true`.
- Во view это поле читается через `request.POST.get('assign_code')`.
- Если флаг есть, вызывается `generate_internal_code()`.
- После сохранения пользователь получает подтверждение с новым кодом.

Почему используются class-based views:
- Они уменьшают количество повторяющегося кода.
- Django сам берет на себя стандартные действия: список, создание, редактирование, удаление.
- Нам остается добавить только бизнес-логику, связанную с кодом и сообщениями.

Особенность создания ОС:
- В `FixedAssetCreateView.form_valid()` код назначается автоматически.
- Нажатие кнопки "Назначить внутренний код" передает флаг `assign_code=true`.

## 8) Быстрый сценарий проверки ЛР

1. Создать 2 подразделения (например `IT`, `BUH`).
2. Добавить 2-3 ОС в `IT`.
3. Проверить коды: `IT_1`, `IT_2`, `IT_3`.
4. Добавить ОС в `BUH`.
5. Проверить код: `BUH_1`.
6. Открыть карточку ОС и убедиться в корректной остаточной стоимости.
7. Проверить поиск/фильтрацию в списке ОС.

Дополнительная проверка логики:
8. Создать второе ОС в том же подразделении и убедиться, что номер увеличился на 1.
9. Открыть форму редактирования и проверить, что код не ломается при сохранении.
10. Изменить подразделение у ОС и проверить, что код при повторном назначении строится уже по новому подразделению.

## 9) Типовые проблемы и решение

1. Ошибка `No module named django`
- Запускать команды через правильный Python 3.11 (см. раздел 2).

2. Сервер не стартует
- Выполнить:
  - `manage.py check`
  - `manage.py makemigrations`
  - `manage.py migrate`

3. Не грузятся стили
- Проверить `STATIC_URL`, `STATICFILES_DIRS` в `Laba7/settings.py`.
